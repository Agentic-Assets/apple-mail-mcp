# apple-mail-mcp Roadmap and Backlog (2026-07-10)

Forward-looking capability and hardening backlog for the plugin. Ground truth at
authoring: `main` at v3.10.0 (41 tools, 11 skills), v3.10.1 open as PR #72
(archiving Human-Sender Screen plus a per-edit hook fix). This file is the
durable "what to build next" reference; the tiny active pointer stays in
[`../todo.md`](../todo.md), and shipped detail lives under
[`../archive/`](../archive/).

Every idea below was checked against the real constraints: all Mail and Calendar
I/O runs through the shared single-flight `run_applescript` lock (draft-first
sends, id-first mutations, bounded reads, dry-run-first deletes, read-only and
draft-safe gating). New-capability ideas were verified against the local
scripting dictionaries (`Mail.sdef`, `Contacts.sdef`) so that "buildable" is
separated from "no macOS API exists" (a documented refusal, like
`respond_to_invitation` and the advisor-only Mail rules).

Priorities are High / Medium / Low by value-to-effort. "Refusal" marks a
capability macOS does not expose; those are listed so we do not re-propose them.

---

## 1. New tools

### High

1. **`get_email_source`** (raw RFC822/MIME by exact id). Mail exposes a
   read-only `source` property on `message`; the tool is already written,
   tested, id-first, and read-only-annotated on branch
   `fix/github-issues-mcp-hardening-20260617`. Port it forward onto the current
   `tools/search/` package layout. Lowest effort, highest certainty: it unblocks
   the unsubscribe helper and header-level inspection. This is the single best
   next build.
2. **Junk marking** (`mark_junk` / `mark_not_junk`, or a `junk` action on
   `update_email_status`). Mail exposes a writable `junk mail status` boolean on
   `message`, currently unused. Same id-first shape as `mark_read` / `flag`;
   fills a real gap (no way today to correct the junk filter or bulk-junk spam).
3. **Colored flags** (a `flag_color` parameter on `update_email_status`). Mail
   exposes `flag index` plus a seven-color highlight enumerator; today only the
   flag boolean is set, so the common color-as-priority convention is invisible
   to agents. Small additive change to an existing tool.

### Medium

4. **Contacts lookup** (`lookup_contact` / `search_contacts`). Contacts.app is
   fully scriptable (`person` with email, phone, organization, note, and `whose`
   filtering). Turns "email Jane about the lease" into an exact address without
   guessing from prior mail, and lets triage answer "is this sender in my
   Contacts" (also a native Mail rule condition the plugin cannot currently
   evaluate). This is net-new surface: it needs its own escaping, timeout, and
   bounded-`whose` discipline mirroring the Mail tools.
5. **`list_signatures`** (read-only). The `signature` element (name plus content)
   is readable on the application. Lets `email-drafting` enumerate valid
   `signature_name` values before composing instead of guessing, and lets
   `email-style-profile` compare captured voice against real saved signatures.
   Signature create/edit is a refusal (read-only element).
6. **Unsubscribe helper** (parsing layer on `get_email_source`). Extract the
   `List-Unsubscribe` header, surface the `mailto:` or `https:` target, and for
   the `mailto:` case optionally hand off to `compose_email` as a draft. The
   `https:` case can only return the link (no browser tool here). Depends on
   tool 1 shipping first.
7. **Robust standard-mailbox resolution** (enhancement to `build_mailbox_ref`).
   Back mailbox resolution with the dictionary shortcuts (`inbox`,
   `junk mailbox`, `sent mailbox`, `drafts mailbox`, `outbox`) instead of fragile
   string names, which drift across Gmail and Exchange naming. Makes
   `manage_trash`, junk marking, and `mailbox-taxonomy` more reliable
   cross-account.

### Low

8. **`reply to` header exposure**. Surface the read-only `reply to` property in
   `get_email_by_id` / `get_email_thread` JSON, for the mailing-list and
   newsletter case where Reply-To differs from From. Cheap, narrow value.

## 2. New skills

### High

1. **Inbound scheduling** (extend `meeting-scheduler`, not a twelfth skill).
   Today the skill triggers on direct user requests. The gap is the reverse:
   an email contains a scheduling ask ("does Tuesday work?"), and the flow should
   be triage, then `check_availability`, then a drafted reply with times.
   Cheapest path is a new "inbound scheduling request" branch plus trigger
   phrases in the existing skill.
2. **Unsubscribe / declutter** (once tool 6 exists). `get_top_senders` plus
   `search_emails` to surface newsletter-heavy senders, `get_email_source` for
   the unsubscribe target, then hand off to `email-archive-cleanup` for the bulk
   move or trash. Natural sibling to the cleanup skill.

### Medium

3. **VIP / important-sender program** (advisor-only, prose output). VIP writes
   are a refusal, so mirror `mail-rules-advisor`: build a recommended VIP list
   from `get_top_senders` and `get_statistics`, and end with steps the user
   applies manually in Mail. Do not invent a fake VIP-write path.
4. **Newsletter / read-later digest**. `constants.py` already carries
   newsletter-detection heuristics; a scheduled run that produces one "here is
   what came in from your subscriptions" digest is a distinct workflow from both
   `inbox-triage` (actionable mail) and `email-management` (sustained cleanup).
5. **End-of-day wrap-up** (cross-surface). Symmetric companion to `inbox-triage`:
   `get_awaiting_reply` for anything due plus tomorrow's `list_events`, framed as
   a single mail-and-calendar review that neither `inbox-triage` nor
   `calendar-operator` currently owns.

### Low

6. **Delegation / out-of-office handoff**. Compose a "here is every open thread
   and who is waiting on what" summary from `get_needs_response`,
   `get_awaiting_reply`, and `export_emails`. Real but niche.

## 3. Enhancements to existing surface

- **`update_email_status`**: add `flag_color` and `mark_junk` / `mark_not_junk`
  actions (tools 2 and 3). Highest-leverage enhancement, reusing the id-first and
  `FILTER_SCAN_DISABLED` machinery already in `manage/status.py`.
- **`get_email_thread` / `get_email_by_id`**: surface `reply to` (tool 8).
- **`build_mailbox_ref`**: dictionary-backed standard-mailbox shortcuts (tool 7).
- **`email-style-profile`**: cross-check captured voice against `list_signatures`
  output (tool 5) rather than inferring closings from Sent text alone.

## 4. Hardening and reliability

### High

1. **Typed AppleScript error at the seam.** `core/applescript.py:run_applescript`
   raises a bare, untyped `Exception` for every non-timeout failure (permission
   denied, Mail.app not running, syntax error, `-1743` automation-denied). There
   is no `AppleScriptError` carrying a code, so the structured error contract
   cannot be honored for these. Introduce one typed exception (at least a raw
   stderr string, ideally a classified code) so the whole codebase can route
   through `serialize_tool_error`. This is the root cause of item 2 and the
   single highest-leverage correctness fix.
2. **Systemic `f"Error: {exc}"` fallback breaks the JSON error contract.**
   Roughly 30 call sites (all four calendar write/destructive tools plus mail
   tools across inbox/compose/manage/analytics) end their `except Exception` in a
   bare string, so an unhandled exception returns unstructured text even when the
   caller passed `output_format="json"`. Fixing item 1 lets these collapse into
   one consistent structured path. Do this as one coordinated pass.

### Medium

3. **No retry/backoff around `run_applescript`.** A single transient failure
   (Mail.app momentarily busy behind the global lock, or a one-off Apple Event
   timeout during a Mail sync) is indistinguishable from a permanent one and is
   never retried. On the large-mailbox stress case this is the most likely
   spurious failure. Add a narrowly-scoped single retry for a specific transient
   error class only (never for user or logic `ToolError`s).
4. **Resolve the `allow_filter_scan` product decision.** `move_email`,
   `update_email_status`, and `manage_trash` still ship a fuzzy filter-based bulk
   mutation escape hatch whose fate has been "runtime migration done, product
   decision open" since 2026-06-29 (see the id-first-search-retirement lane).
   Bulk mutate-by-fuzzy-filter is safety-relevant and still reachable in
   production; decide keep-with-guardrails versus remove, and write it down.
5. **`manage_drafts` mailbox-scoping undercount.** A live note (native-reply
   handoff, 2026-06-30) recorded `drafts list --account "TU - Cayman"` returning
   `count=0` while AppleScript directly saw 208 drafts, filed as an aside and
   never promoted to a tracked bug. Likely a real silent correctness bug (hiding
   drafts). Reproduce, root-cause, and fix or document.

### Low

6. **EventKit silent field-level data loss.** `calendar_core/eventkit.py` has
   about ten `except Exception: return None` guards around individual field reads
   (default calendar name, participant email, alarm offset, attendee status).
   Fail-open is reasonable for an optional accelerator, but only `fetch_window`'s
   per-event failure reaches `errors[]`; the smaller field-level swallows never
   surface, so data loss on this path is invisible to the caller. Surface a
   count or note when field reads are dropped.
7. **Brand-voice em-dash sweep.** 13 em dashes across 10 agent-facing
   `@mcp.tool` docstrings (analytics/statistics, compose/forward, compose/manage,
   compose/reply, compose/send, inbox/accounts, inbox/mailboxes, manage/sync,
   search/by_id, smart_inbox/needs_response), plus one in `server.json` (a shipped
   manifest). Decide whether the banned-em-dash policy covers user-facing copy
   only or also code comments (37 `.py` files contain em dashes overall), then
   sweep and rebuild artifacts. Tracked as a deferred brand-voice pass.

## 6. Documented refusals (no macOS API; do not re-propose)

- **VIP list management**: `sender is VIP` exists only as a Mail rule condition;
  there is no VIP class to read or write. Advisor-only, like Mail rules.
- **Signature create/edit**: the `signature` element is read-only. `list_signatures`
  (read) is fine; writes are not.
- **Smart mailbox / saved-search reads**: no smart-mailbox class exists in the
  dictionary. A saved search can only be emulated at the skill layer (store and
  replay a query), never read as a Mail object.
- **Attachment content search** (full text inside PDFs and docs): `mail attachment`
  exposes only name, MIME type, size, and downloaded state. Real full-text search
  would require shelling out to Spotlight / `mdfind` on saved files, a new
  subprocess surface outside the `run_applescript`-only doctrine. Out of scope
  unless that doctrine is revisited.
- **RSVP / invitation delivery guarantees**: no public macOS API can RSVP, and
  EventKit participant status is read-only. Already documented in the calendar
  tools and skills.

## 7. If we build three things next

1. **Port `get_email_source` forward** (finished work sitting idle on a branch).
2. **Junk plus colored-flag actions on `update_email_status`** (writable,
   dictionary-confirmed, zero new architecture, reuses existing code).
3. **The typed-`AppleScriptError` error-contract pass** (hardening items 1 and 2)
   before more tools are added, so new surface inherits a consistent contract.

Contacts lookup is the biggest net-new capability (address resolution and sender
enrichment) and is the natural fourth once the error-contract cleanup lands.
