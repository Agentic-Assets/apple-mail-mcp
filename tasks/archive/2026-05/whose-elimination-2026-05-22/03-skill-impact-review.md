## Cross-cutting themes

1. **`allow_full_scan` is in 5 files, ~10 occurrences.** Phase A removal means a coordinated rewrite. The replacement framing (structured "scanned N, found 0, narrow your query" errors) doesn't yet have prose anywhere — every site currently says "do not call without user confirmation." Authors need a single canonical sentence to copy across all 5 SKILL.md files.

2. **The pre-flight block is duplicated verbatim across 4 SKILL.md files** (`apple-mail-operator`, `email-management`, `email-archive-cleanup`, `inbox-triage`) — identical L10–20. Phase A is the right moment to single-source it (a `plugin/skills/references/large-inbox-rules.md` or `_includes`-style snippet), otherwise Phase B doc-sync becomes a 4-way string match.

3. **`recent_days` ladders only appear explicitly once** (`email-style-profile:27`). Most other skills teach "start small, widen on demand" as a one-liner, not a ladder. So Phase B obsoletes one explicit ladder + tones down ~8 "start narrow" sentences across skills.

4. **`scan_cap` / "bounded slice" phrasing is rare in skills.** Only `tools/CLAUDE.md:23` ("bounded newest-message slices `messages 1 thru N`") names the pattern. SKILL.md files talk about `max_emails`, `recent_days`, `limit`, `max_results` — they never expose `scan_cap` to agents. **Good news for Phase A**: making it the invisible default requires almost no skill edits, only the tools/CLAUDE.md sentence.

5. **`inbox_dashboard` "rescue path" appears in 5 files** (4 SKILL.md pre-flights + 1 dedicated subsection in `apple-mail-operator:83-91` + 1 line in `email-management:160`). Phase B reframes all 5.

6. **Tool-selection performance hierarchies live mainly in `email-management:152–174`.** That's the only deep "this is fast / this is slow / this is costly" table. `apple-mail-operator:57-65` has a smaller decision tree without perf claims. Phase B flattens the `email-management` table substantially.

7. **Cross-references are healthy.** All `[[skill-name]]` and "see X skill" links resolve to actual directories. No broken sibling routing under the planned refactor.

8. **Standalone-compose safeguard mentions are correct under both phases** — verified at: `apple-mail-operator:79`, `email-management:141`, `email-drafting:28/55/82-83`, `email-style-profile:44`, `tools/CLAUDE.md:43`. All wording aligns with v3.1.10. No edits needed for this category.

## Recommended doc-sync sequencing

- **Phase A: skills go in the same PR as code, not one PR behind.** Reason: the `allow_full_scan` parameter removal is a breaking change to the *user-visible* MCP surface. If skills still document a removed parameter, the model will call it, the tool will error, and the structured "narrow your query" message becomes noise the model is unprepared for. Land the structured-error UX, the `core.bounded_inbox_scan()` helper, the CI lint banning raw `whose`, **and** the 5 SKILL.md updates + `tools/CLAUDE.md` update in one PR. Pre-flight block deduplication can ship as a follow-up unless the same PR already touches all four files.
- **Phase B: skills lag code by one PR, intentionally.** Reason: Envelope Index migration is a perf/internal change, not a behavior change for the agent. Existing skill guidance ("prefer narrow queries") stays *correct, just unnecessary* the day the read tools migrate. Ship the Envelope Index migration first, validate against the 24k-inbox baseline, then issue a single skills-only PR that (a) reframes `inbox_dashboard` from "rescue path" → "one-shot summary," (b) flattens the `email-management` Tool Selection Guidelines table, (c) deletes the `email-style-profile` ladder, (d) trims the pre-flight blocks to mutation-only. This keeps the perf PR scoped and lets the skills PR ride on real benchmark numbers.
