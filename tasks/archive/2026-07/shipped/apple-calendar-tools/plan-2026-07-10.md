# Apple Calendar tools: implementation plan (target release 3.10.0)

Branch `feat/apple-calendar-tools`. Evidence base:
[`research-report-2026-07-10.md`](research-report-2026-07-10.md) (cited below as RR with a
section number). Style contract for everything produced under this plan: no em dashes,
and none of the banned words from `policies/brand-voice.md`, in code comments,
docstrings, skills, manifests, and CHANGELOG alike.

---

## 1. MCP tools vs skills: verdict

**Verdict: MCP tools are the bridge layer; skills are workflow guidance on top. Both
ship. The owner's question "would MCP tools give us more functionality?" has a direct
answer: yes, materially, and the reference skill is the proof by counterexample.**

What tools provide that a script-based skill cannot (each point maps to a concrete
failure in the reference skill, RR section 2):

1. **Structured JSON in and out.** Typed parameters with defaults and constraints on the
   wire, one JSON object per event on the way back. The reference skill emits three
   mutually inconsistent unescaped text formats; a `|` inside an event title can shift a
   UID under a later delete (RR 2.4, 6.5).
2. **Input escaping and injection safety by construction.** Every user string passes
   `core.escape_applescript()`; datetimes are validated integers, never interpolated
   strings; RRULEs pass an allowlist grammar. A skill invoking `osascript` from prose
   cannot guarantee any of this on every call.
3. **Safe-mode gating enforced in code.** `--read-only` removes calendar write tools from
   the registry; `--draft-safe` blocks deletes and attendee sends inside the tool body.
   A skill can only ask the model to be careful; a tool refuses with a structured error.
4. **Bounded scans enforced in code.** The window producer refuses unbounded reads before
   any AppleScript runs, with hard caps a caller cannot exceed. The reference skill lets
   any caller request 3650 days across every calendar with no result cap (RR 2.2).
5. **Serialization and timeout discipline.** All osascript calls flow through
   `core.run_applescript` (single-flight lock, 120 s default timeout, sanitized output).
   The reference skill has no timeout anywhere; on this machine a pending TCC prompt
   hangs forever without one (RR 5).
6. **Testability.** Tools get 1000+ mocked tests, mypy `--strict`, lint gates, and
   manifest parity checks in this repo's CI. Shell-script skills get none of that.
7. **Distribution.** Tools ship identically on all four install surfaces (Claude Code,
   Cowork `.plugin`, Claude Desktop `.mcpb`, Codex). A skill with bundled shell scripts
   does not reach the `.mcpb` surface at all.

Skills still carry real weight: tool selection, the find-slot then create workflow,
timezone discipline, destructive red lines, and honest documentation of the two platform
gaps (invitations, RSVP). That split (tools enforce, skills teach) is exactly the
existing mail architecture. Section 6 specifies the two new skills.

## 2. Engine architecture

**Decision: hybrid. AppleScript via `core.run_applescript()` is the guaranteed engine
for every operation on every surface. EventKit via PyObjC is an optional, read-only fast
path that activates only when its dependency is installed and Calendars full access is
already granted. All writes go through AppleScript in 3.10.0.**

Rationale from evidence:

- AppleScript rides the Apple Events Automation TCC grant that this repo's Mail tools
  already require, so it works on all four install surfaces including the Claude Desktop
  `.mcpb`, where EventKit is denied synchronously with no prompt because the host app
  lacks the Calendars usage string (RR 4.4: claude-code#63032, codex#21228).
- EventKit reads are two to three orders of magnitude faster than Calendar.app
  AppleScript reads (0.13 s vs 60 to 112 s on community benchmarks, RR 4.6), and EventKit
  expands recurring occurrences correctly where AppleScript window predicates do not.
  On terminal-attributed hosts (Claude Code, Codex CLI), the grant is obtainable, so the
  fast path is worth shipping behind detection.
- The live probe (RR 5) showed both paths blocked on this machine until a human answers
  the consent prompts once, and showed that a bare-interpreter EventKit access request
  can hang. Therefore the server must never trigger the EventKit consent prompt from a
  tool call; the prompt is requested only by an explicit CLI command run by a human.
- A signed, notarized, self-disclaiming Swift helper is the only EventKit-everywhere
  route and is out of scope for a pure-Python release (RR 4.4). Tracked as a forward
  item, alongside watching claude-code#63032 for an Anthropic-side Info.plist fix.

### 2.1 Engine seam

`calendar_core/engine.py` defines a small `CalendarEngine` Protocol consumed by every
tool: `list_calendars()`, `fetch_events(window, calendars, ...)`,
`fetch_events_by_id(ids, window, calendar)`, `create_event(spec)`, `update_event(...)`,
`delete_events(...)`, `create_calendar(...)`, `rename_calendar(...)`,
`delete_calendar(...)`, `count_events(calendar)`.

Engine selection (`get_engine()`), evaluated lazily per call so tests can patch:

1. `APPLE_MAIL_CALENDAR_ENGINE` env override: `applescript` | `eventkit` | `auto`
   (default `auto`).
2. Under `auto`: use the EventKit read engine iff `EventKit` is importable and
   `EKEventStore.authorizationStatusForEntityType_(EKEntityTypeEvent)` already reports
   full access. `notDetermined`, denied, restricted, write-only, or missing dependency
   all fall back silently to AppleScript. No tool call ever invokes
   `requestFullAccessToEvents...`.
3. Writes always dispatch to the AppleScript engine in 3.10.0 regardless of selection
   (single write path = single injection surface, single test matrix, and span
   limitations documented in one place). Every read response echoes `"engine"` so the
   active path is always visible.

### 2.2 Dependency plan

- `pyproject.toml`: **no new required dependencies.** Add an optional extra:
  `[project.optional-dependencies] eventkit = ["pyobjc-framework-EventKit>=10,<13"]`
  (pulls `pyobjc-core` and `pyobjc-framework-Cocoa`; measured install 1.6 s / a few MB,
  RR 5.1). Install command documented as `pip install 'mcp-apple-mail[eventkit]'`.
- `calendar_core/eventkit.py` imports EventKit inside a guarded helper
  (`importlib.util.find_spec` check first); the module itself always imports cleanly on
  CI (Ubuntu, no PyObjC).
- **MCPB bundle: do not bundle PyObjC.** EventKit is dead inside Claude Desktop today
  (RR 4.4), so bundling is pure weight; the `.mcpb` runs the AppleScript engine. No
  change to `apple-mail-mcpb/build-mcpb.sh` beyond version and tool metadata.
- TCC grant flow: new CLI command `apple-mail calendar-grant` performs the EventKit
  full-access request with a 30 s runloop pump, for humans at a terminal. Tool-side TCC
  failures surface as `CALENDAR_ACCESS_DENIED` with pane-specific remediation, and
  AppleScript first-use timeouts mention the Automation pane (RR 5.3, 6.6).

### 2.3 Timezone strategy per engine

Requirement 7 is satisfied in Python, not in AppleScript: requested zoned times are
converted with `zoneinfo` to host-local wall-clock integers before script interpolation,
so the created event lands on the correct absolute instant regardless of the Mac's zone.
Reads emit numeric date components from AppleScript (never locale-dependent
`as string` coercion), composed into ISO 8601 with offsets in Python, and returned in
both the requested zone and UTC. The stored per-event timezone property is an
EventKit-write capability and stays on the forward queue (documented in tool docs and
the calendar-operator skill).

## 3. Final tool surface (10 new tools; 31 + 10 = 41 total)

Shared conventions: every tool returns `str`; every read supports
`output_format: str = "json" | "text"` with **json as the default** (requirement 9;
deliberate deviation from mail's text default, noted in tool docstrings); every tool
takes `timeout: int | None = None` threaded into the engine; structured errors always
return the JSON envelope even under `output_format="text"`; every read response includes
`engine`, `resolved_timezone`, and (when applicable) `truncated` + `next_offset`.
Calendar name params are fuzzy-resolved for reads and create targets only
(exact case-insensitive match wins; multiple candidates return
`AMBIGUOUS_CALENDAR_SELECTOR` with `remediation.candidates`; no match returns
`CALENDAR_NOT_FOUND` with close candidates). Event ids are validated for shape
(non-empty, <=512 chars, no `|||`, no control characters) before use.

### 3.1 `list_calendars` (READ_ONLY_TOOL_ANNOTATIONS)

```python
def list_calendars(output_format: str = "json", timeout: int | None = None) -> str
```

Returns every calendar: `calendar_id` (AppleScript calendar `uid` when exposed, else the
exact name, with `id_kind` reported), `name`, `writable`, `account`, `is_default`.
Top-level: `default_calendar` (resolution order: `DEFAULT_CALENDAR` env, then the
engine's default, then none), `engine`, `eventkit_available` (diagnostic for the fast
path). No caps needed; calendar counts are tiny. Errors: `CALENDAR_ACCESS_DENIED`.

### 3.2 `list_events` (READ_ONLY_TOOL_ANNOTATIONS)

Covers list, today, upcoming, and search in one tool (query param replaces a separate
search tool).

```python
async def list_events(
    calendar: str | None = None,        # fuzzy; None = all calendars, capped fan-out
    calendars: list[str] | None = None, # explicit multi-calendar scope (mutually exclusive with calendar)
    start: str | None = None,           # ISO 8601 (offset-aware or naive + timezone)
    end: str | None = None,
    days_back: float = 0.0,
    days_ahead: float = 7.0,            # bounded default: next 7 days
    timezone: str | None = None,        # IANA zone for interpretation and output
    query: str | None = None,           # case-insensitive substring over title/location/notes, matched in Python
    include_all_day: bool = True,
    expand_recurring: bool = True,
    output_format: str = "json",
    offset: int = 0,
    limit: int | None = None,           # min(limit, EVENT_RETURN_CAP)
    timeout: int | None = None,
) -> str
```

Hard caps: window width <= `MAX_WINDOW_DAYS` (370); returns <= `EVENT_RETURN_CAP` (200)
with paging; occurrence expansion <= `OCCURRENCE_SCAN_CEILING` (750); unscoped fan-out
<= `MAX_CALENDARS_PER_QUERY` (20) calendars with per-calendar errors collected into
`calendar_errors` (partial results over total failure). Zero-width or absent window
(both day params 0 and no absolute window) refuses with `UNBOUNDED_CALENDAR_SCAN`.

Event row shape: `event_id`, `calendar`, `calendar_id`, `title`, `start`, `end`
(ISO 8601 with offset), `start_utc`, `end_utc`, `all_day`, `location`, `notes_preview`
(capped 280 chars), `url`, `recurring` (bool), `recurrence_rule`, `occurrence_date`
(for expanded occurrences), `expansion` (`"native" | "python" | "unsupported_rrule"`),
`status`, `attendees` (read-only: name, email, participation status).

AppleScript engine detail: primary bounded `whose start date` pass, plus a second
bounded recurring-master pass (`start date` within `RECURRING_LOOKBACK_DAYS` before the
window and `recurrence` non-empty) expanded in Python by `calendar_core/recurrence.py`
against the allowlisted RRULE grammar; masters with unsupported rules are returned
flagged, never silently dropped (RR 4.6, 7.4).

Errors: `UNBOUNDED_CALENDAR_SCAN`, `CALENDAR_WINDOW_TOO_WIDE`,
`CALENDAR_WINDOW_TOO_DENSE`, `INVALID_EVENT_WINDOW`, `INVALID_TIMEZONE`,
`CALENDAR_NOT_FOUND`, `AMBIGUOUS_CALENDAR_SELECTOR`, `CALENDAR_ACCESS_DENIED`.

### 3.3 `get_events_by_id` (READ_ONLY_TOOL_ANNOTATIONS)

```python
def get_events_by_id(
    event_ids: list[str],               # 1..MAX_EVENT_IDS_PER_CALL (25)
    calendar: str | None = None,        # strongly recommended; bounds the scan to one calendar
    start: str | None = None,           # optional explicit lookup window
    end: str | None = None,
    days_back: float = 30.0,            # default bounded lookup window when no absolute window
    days_ahead: float = 90.0,
    timezone: str | None = None,
    output_format: str = "json",
    timeout: int | None = None,
) -> str
```

Full detail per event (untruncated notes, alarms as minutes-before list, full attendee
list). The UID lookup is always window-bounded: this is the fix for the reference
skill's worst anti-pattern (unbounded `whose uid` scans, RR 2.4). Returns
`{"events": [...], "missing": [ids...]}` rather than failing the whole call on one
missing id. Errors: `INVALID_EVENT_ID`, window/timezone/calendar errors as above,
`WHOSE_ID_LIST_TOO_LARGE` semantics enforced by chunking ids 25 per osascript call.

### 3.4 `check_availability` (READ_ONLY_TOOL_ANNOTATIONS)

```python
async def check_availability(
    start: str,                         # required, ISO 8601
    end: str,                           # required; width <= AVAILABILITY_MAX_WINDOW_DAYS (62)
    timezone: str | None = None,
    calendars: list[str] | None = None, # None = all calendars (capped fan-out)
    slot_minutes: int = 30,             # 5..480
    working_hours_start: str = "09:00",
    working_hours_end: str = "17:00",
    weekdays_only: bool = True,
    max_slots: int = 20,                # 1..50
    ignore_free_availability: bool = True,  # events marked availability=free do not block
    include_busy_blocks: bool = True,
    output_format: str = "json",
    timeout: int | None = None,
) -> str
```

Answers both "am I busy at X" (busy blocks overlapping the window) and "find a free
slot" (free slots of `slot_minutes` inside working hours). All folding happens in Python
over one bounded fetch (RR 4.3: no native free-busy API exists on any engine). Returns
`{"busy": [{start, end, title, calendar, event_id}], "free_slots": [{start, end}],
"resolved_timezone", "window", "engine"}`. Errors: `UNBOUNDED_CALENDAR_SCAN` (missing
start/end), `CALENDAR_WINDOW_TOO_WIDE` (62-day cap), `INVALID_SLOT_PARAMS`,
`INVALID_TIMEZONE`, calendar resolution errors.

### 3.5 `create_event` (WRITE_TOOL_ANNOTATIONS)

```python
def create_event(
    title: str,
    start: str,                         # ISO 8601
    end: str | None = None,             # exactly one of end / duration_minutes; all_day may omit both (1 day)
    duration_minutes: int | None = None,
    calendar: str | None = None,        # fuzzy; falls back to DEFAULT_CALENDAR env, then engine default
    timezone: str | None = None,
    all_day: bool = False,
    location: str | None = None,
    notes: str | None = None,
    url: str | None = None,
    alarms_minutes_before: list[int] | None = None,  # <= MAX_ALARMS_PER_EVENT (5), each 0..40320
    recurrence: str | None = None,      # RRULE, allowlist-validated
    attendees: list[str] | None = None, # email addresses; gated, see below
    send_invitations: bool = False,     # must be True whenever attendees is non-empty
    on_conflict: str = "warn",          # "warn" | "block" | "allow"
    output_format: str = "json",
    timeout: int | None = None,
) -> str
```

Conflict detection (requirement 10) is in-line and on by default: one bounded query over
the new event's own `[start, end]` on the target calendar; `warn` creates and reports
`conflicts[]`, `block` refuses with `EVENT_CONFLICT` writing nothing, `allow` skips the
check (RR 6.4). Attendee gating (requirements 1 and safety): `attendees` without
`send_invitations=True` refuses with `INVITE_SEND_REQUIRES_CONFIRM`; under `--draft-safe`
or `--read-only` any attendee use refuses with `INVITE_SEND_BLOCKED`; attendee count
capped at `MAX_ATTENDEES` (50); response always carries
`"invitation_delivery": "platform_dependent"` with a note that macOS offers no public
API that guarantees invitation transmission, plus the `.ics`-via-Mail alternative
(RR 4.2, 7.2). Returns `created: true`, `event_id`, echo of what was written,
`has_conflicts` + `conflicts[]`, `resolved_timezone`.

Errors: `CALENDAR_WRITE_BLOCKED`, `CALENDAR_NOT_FOUND`, `AMBIGUOUS_CALENDAR_SELECTOR`,
`CALENDAR_READ_ONLY`, `INVALID_EVENT_WINDOW`, `INVALID_TIMEZONE`,
`INVALID_RECURRENCE_RULE`, `INVALID_ALARM`, `EVENT_CONFLICT`,
`INVITE_SEND_REQUIRES_CONFIRM`, `INVITE_SEND_BLOCKED`, `TOO_MANY_ATTENDEES`.

### 3.6 `batch_create_events` (WRITE_TOOL_ANNOTATIONS)

```python
def batch_create_events(
    events: list[dict],                 # 1..BATCH_CREATE_CAP (25); items: title, start, end|duration_minutes,
                                        # optional all_day/location/notes/url/alarms_minutes_before/timezone
    calendar: str | None = None,        # one target calendar for the whole batch
    timezone: str | None = None,        # batch default zone
    on_conflict: str = "warn",
    dry_run: bool = False,              # True = validate + conflict-preview all items, write nothing
    output_format: str = "json",
    timeout: int | None = None,
) -> str
```

Requirement 4a (create a series of one-off events from a list). Items may not contain
`attendees` or `recurrence` (per-item validation error; keeps batch personal-blocking
only and the invite path single-event). All items validate before any write; creation is
then per-item with partial results (`created[]` with ids, `failed[]` with reasons).
Errors: `BATCH_TOO_LARGE`, per-item `INVALID_EVENT_WINDOW`/`INVALID_ALARM`/etc. in the
item report, plus the create-tool calendar and mode errors.

### 3.7 `update_event` (IDEMPOTENT_WRITE_TOOL_ANNOTATIONS)

```python
def update_event(
    event_id: str,
    calendar: str | None = None,        # lookup hint (bounds the scan)
    start_hint: str | None = None,      # optional lookup window hint around the event's known date
    title: str | None = None,           # PATCH semantics: None = leave unchanged
    start: str | None = None,
    end: str | None = None,
    duration_minutes: int | None = None,
    timezone: str | None = None,
    all_day: bool | None = None,
    location: str | None = None,
    notes: str | None = None,
    url: str | None = None,
    alarms_minutes_before: list[int] | None = None,  # replaces the alarm set
    clear_alarms: bool = False,
    recurrence: str | None = None,      # replaces the rule
    clear_recurrence: bool = False,
    span: str | None = None,            # required when the target is recurring
    attendees: list[str] | None = None, # same gating as create_event
    send_invitations: bool = False,
    on_conflict: str = "warn",          # applies when start/end change
    dry_run: bool = False,              # preview resolved target + field diff
    output_format: str = "json",
    timeout: int | None = None,
) -> str
```

ID-first only; there is no fuzzy target parameter at all, by construction (RR 6.5).
Recurring targets require explicit `span`; the AppleScript engine supports only
`span="all_occurrences"` and returns `RECURRING_SPAN_UNSUPPORTED` for
`this_occurrence`/`future_occurrences` with remediation naming the EventKit write engine
as the future unlock (RR 8, delete row). Errors: `EVENT_NOT_FOUND`, `INVALID_EVENT_ID`,
`RECURRING_SPAN_REQUIRED`, `RECURRING_SPAN_UNSUPPORTED`, plus create-tool errors.

### 3.8 `delete_events` (DESTRUCTIVE_TOOL_ANNOTATIONS)

One tool for single and bulk delete (requirement 4b): exact ids only, never a query.

```python
def delete_events(
    event_ids: list[str],               # 1..100 ids, exact only
    calendar: str | None = None,        # lookup hint
    start: str | None = None,           # optional lookup window
    end: str | None = None,
    days_back: float = 30.0,
    days_ahead: float = 90.0,
    span: str | None = None,            # required if any target is recurring
    dry_run: bool = True,               # DEFAULT TRUE: preview resolved targets first
    max_deletes: int = 20,              # BULK_DELETE_DEFAULT_MAX; absolute ceiling 100
    output_format: str = "json",
    timeout: int | None = None,
) -> str
```

Gates: under `--draft-safe`, refuses with `CALENDAR_DELETE_BLOCKED` unless the operator
launched with `CALENDAR_ALLOW_DESTRUCTIVE=1` (env-only unlock; never a tool argument).
Under `--read-only` the tool is removed from the registry, with an internal
`CALENDAR_WRITE_BLOCKED` backstop for the CLI path. `dry_run=True` resolves every id to
`{event_id, title, start, calendar, recurring}`; any unresolved id fails the whole
preview with `EVENT_NOT_FOUND` (a typo never partially deletes). Recurring ids without
`span` refuse with `RECURRING_SPAN_REQUIRED`; AppleScript engine accepts only
`all_occurrences` (whole-series delete is the platform behavior, stated in the response).
Caps: `TOO_MANY_DELETES` above `max_deletes` or the 100 ceiling; ids chunked 25 per
osascript call. Deleted-event summaries are captured before deletion for the
confirmation payload.

### 3.9 `manage_calendars` (DESTRUCTIVE_TOOL_ANNOTATIONS; multiplexed like `manage_trash`)

Requirement 5, one tool instead of three.

```python
def manage_calendars(
    action: str,                        # "create" | "rename" | "delete"
    name: str | None = None,            # create: new name; rename/delete: EXACT existing name
    calendar_id: str | None = None,     # rename/delete: preferred exact selector
    new_name: str | None = None,        # rename
    dry_run: bool = True,               # delete only: preview cascade event count
    confirm_delete_calendar: bool = False,
    force_nonempty: bool = False,
    output_format: str = "json",
    timeout: int | None = None,
) -> str
```

`create` and `rename` are writes (allowed under `--draft-safe`, mirroring
`create_mailbox`); `delete` is the most dangerous operation in the surface (cascade) and
requires, in order: a `dry_run` preview reporting the exact event count, then
`dry_run=False` + `confirm_delete_calendar=True`, plus `force_nonempty=True` when the
count is non-zero. Fuzzy matching is disabled for `rename` and `delete` (exact name or
`calendar_id` only). `delete` under `--draft-safe` refuses with
`CALENDAR_DELETE_BLOCKED` (same env unlock as 3.8). Errors:
`CALENDAR_CONFIRMATION_REQUIRED`, `CALENDAR_ALREADY_EXISTS` (create collision),
`CALENDAR_NOT_FOUND`, `CALENDAR_READ_ONLY`, `CALENDAR_DELETE_BLOCKED`,
`CALENDAR_WRITE_BLOCKED`.

### 3.10 `respond_to_invitation` (READ_ONLY_TOOL_ANNOTATIONS; documented-refusal shim)

```python
def respond_to_invitation(
    event_id: str | None = None,
    response: str | None = None,        # "accept" | "decline" | "tentative"
    output_format: str = "json",
) -> str
```

Always returns `CALENDAR_RSVP_UNSUPPORTED` with remediation naming the platform reality

asked for on requirement 8.

## 4. Safety spec (adopted from phase-1 safety design, adjusted)

Adjustments from RR section 7: engine-neutral contract applied to the hybrid engine;
attendee handling gated as outward-facing and documented as non-guaranteed delivery;
`RECURRING_SPAN_UNSUPPORTED` added for the AppleScript span limitation; consolidated
tool names in the matrix.

### 4.1 `CALENDAR_BOUNDS` (new dict in `constants.py`)

```python
CALENDAR_BOUNDS = {
    "MAX_WINDOW_DAYS": 370,             # hard cap on window width (year + 2 weeks slack)
    "EVENT_RETURN_CAP": 200,            # max events returned per call; page via offset
    "OCCURRENCE_SCAN_CEILING": 750,     # recurring-expansion hard stop per call
    "AVAILABILITY_MAX_WINDOW_DAYS": 62, # free-busy / slot-finding window cap
    "BATCH_CREATE_CAP": 25,             # batch_create_events items per call
    "BULK_DELETE_DEFAULT_MAX": 20,      # delete_events default max_deletes
    "BULK_DELETE_CEILING": 100,         # absolute delete ceiling per call
    "MAX_ATTENDEES": 50,                # attendee cap per event
    "MAX_ALARMS_PER_EVENT": 5,
    "MAX_EVENT_IDS_PER_CALL": 25,       # get_events_by_id input cap and per-osascript chunk size
    "MAX_CALENDARS_PER_QUERY": 20,      # unscoped fan-out cap
    "DEFAULT_UPCOMING_DAYS": 7,         # list_events bounded default window
    "UID_LOOKUP_BACK_DAYS": 30,         # default id-lookup window
    "UID_LOOKUP_AHEAD_DAYS": 90,
    "RECURRING_LOOKBACK_DAYS": 400,     # recurring-master second-pass horizon (AppleScript engine)
    "NOTES_PREVIEW_CHARS": 280,
}
```

One sanctioned window producer, `bounded_calendar_window(...)` in
`calendar_core/window.py`, is the only issuer of `CalendarWindow` tokens (the
`bounded_inbox_scan` analogue); engines accept only issued tokens, so no tool can
smuggle an unbounded scan.

### 4.2 Mode gating matrix

| Tool / path | default | `--draft-safe` | `--read-only` |
|---|---|---|---|
| `list_calendars`, `list_events`, `get_events_by_id`, `check_availability` | ALLOW (bounded) | ALLOW | ALLOW |
| `create_event` / `update_event` / `batch_create_events` (no attendees) | ALLOW | ALLOW | BLOCK (removed + backstop) |
| `manage_calendars(action="create" or "rename")` | ALLOW | ALLOW | BLOCK (removed + backstop) |
| create/update with `attendees` | RESTRICT: requires `send_invitations=True` | BLOCK `INVITE_SEND_BLOCKED` | BLOCK (removed) |
| `delete_events` | RESTRICT: `dry_run=True` default, `max_deletes`, `span` for recurring | BLOCK `CALENDAR_DELETE_BLOCKED` (env unlock: `CALENDAR_ALLOW_DESTRUCTIVE=1`) | BLOCK (removed) |
| `manage_calendars(action="delete")` | RESTRICT: dry-run + confirm + force_nonempty, exact selector only | BLOCK `CALENDAR_DELETE_BLOCKED` (env unlock) | BLOCK (removed) |
| `respond_to_invitation` | BLOCK `CALENDAR_RSVP_UNSUPPORTED` | same | same (stays registered) |

`server.py` additions:
`CALENDAR_WRITE_TOOLS = ("create_event", "update_event", "batch_create_events", "manage_calendars")`
and `CALENDAR_DESTRUCTIVE_TOOLS = ("delete_events",)`; `__main__.py` removes both tuples
under `--read-only` exactly like `SEND_TOOLS`. Every write/delete tool keeps an internal
`_server.READ_ONLY` / `_server.DRAFT_SAFE` guard (the `_send_blocked` pattern) because
the CLI bypasses registry removal. Env additions to `server.py`: `DEFAULT_CALENDAR`
(read lazily by tools) and `CALENDAR_ALLOW_DESTRUCTIVE` (read at startup).

### 4.3 Structured error codes (all via `ToolError` / `serialize_tool_error`)

| Code | Trigger |
|---|---|
| `UNBOUNDED_CALENDAR_SCAN` | Event query with no window and no bounded default in effect |
| `CALENDAR_WINDOW_TOO_WIDE` | Window width > 370 days (62 for availability) |
| `CALENDAR_WINDOW_TOO_DENSE` | Occurrence expansion exceeds 750 in the bounded window |
| `INVALID_EVENT_WINDOW` | start >= end, malformed datetime, or conflicting window forms |
| `INVALID_TIMEZONE` | Unknown or malformed IANA zone |
| `INVALID_EVENT_ID` | Malformed event id (shape validation) |
| `INVALID_RECURRENCE_RULE` | RRULE outside the allowlist grammar |
| `INVALID_ALARM` | Alarm count > 5 or minutes outside 0..40320 |
| `INVALID_SLOT_PARAMS` | slot_minutes/working-hours/max_slots out of range |
| `EVENT_NOT_FOUND` | An id fails to resolve (fails a whole delete preview) |
| `CALENDAR_NOT_FOUND` | No calendar matches after fuzzy resolution (candidates listed) |
| `AMBIGUOUS_CALENDAR_SELECTOR` | Fuzzy name matched more than one calendar (candidates listed) |
| `CALENDAR_ALREADY_EXISTS` | `manage_calendars(action="create")` name collision |
| `CALENDAR_READ_ONLY` | Write/delete targeted a subscribed or system calendar |
| `CALENDAR_WRITE_BLOCKED` | Any write while `READ_ONLY` (belt-and-suspenders) |
| `CALENDAR_DELETE_BLOCKED` | Any delete while `DRAFT_SAFE` without the env unlock |
| `INVITE_SEND_REQUIRES_CONFIRM` | `attendees` without explicit `send_invitations=True` |
| `INVITE_SEND_BLOCKED` | Attendee use under `DRAFT_SAFE`/`READ_ONLY` |
| `TOO_MANY_ATTENDEES` | Attendee list > 50 |
| `BATCH_TOO_LARGE` | Batch items > 25 |
| `TOO_MANY_DELETES` | Ids > `max_deletes` or > 100 ceiling |
| `CALENDAR_CONFIRMATION_REQUIRED` | Calendar delete without confirm/force flags |
| `RECURRING_SPAN_REQUIRED` | Mutation on a recurring id without explicit `span` |
| `RECURRING_SPAN_UNSUPPORTED` | Non-whole-series `span` on the AppleScript engine |
| `EVENT_CONFLICT` | `on_conflict="block"` and an overlap exists |
| `CALENDAR_RSVP_UNSUPPORTED` | `respond_to_invitation` (always) |
| `CALENDAR_ACCESS_DENIED` | TCC denial (EventKit status or AppleScript `-1743`), with pane + access level in remediation |

Reused, not duplicated: the mail envelope shape, and `TARGET_SELECTOR_DEPRECATED` stays
reserved (calendar mutations expose no fuzzy selectors at all, so it cannot fire today).

### 4.4 Bounded-read contract, injection rules, identifier doctrine, TCC UX

Adopted verbatim from RR sections 6.1, 6.5, and 6.6: mandatory windows with the caps
above; `escape_applescript` on every user string; integer-validated date interpolation;
allowlisted RRULE grammar; `sanitize_pipe_delimited_field` plus Python-side UID shape
checks on the internal row protocol (which is never exposed to callers); JSON-only tool
output; ID-first mutations with no fuzzy destructive targets; span-required recurring
mutations; fuzzy calendar resolution never applied to destructive targets;
`CALENDAR_ACCESS_DENIED` remediation naming the exact System Settings pane, and
first-use AppleScript timeouts mentioning the Automation pane (live-probe finding: a
pending Automation prompt hangs silently rather than erroring, RR 5).

## 10. Open risks

1. **TCC state on the dev machine.** Both grants are unresolved (RR 5); until a human
   answers the prompts, no live verification or real timing is possible. Mitigated by
   the timeout discipline and the `CALENDAR_ACCESS_DENIED` UX; the risk is schedule,
   not safety.
2. **AppleScript read performance on dense calendars is unmeasured here.** Community
   numbers (60 to 112 s) are for unbounded scans on other machines; our bounded windows
   plus caps may still be slow on a years-deep iCloud calendar. `CALENDAR_BOUNDS` is
   centralized precisely so live results can retune caps in one edit; the EventKit fast
   path is the structural fix on terminal hosts.
3. **Attendee behavior is account-type-dependent.** Evidence says script-attached
   attendees do not send invitations and may be dropped on sync (RR 4.2); some account
   types may behave differently. Shipped gated, confirmed, and documented as
   non-guaranteed; the skill teaches the `.ics`-via-Mail alternative. Residual risk is
   user surprise, bounded by the explicit confirm flag and response disclosure.
4. **Recurring-series semantics under AppleScript.** Whole-series-only span, shared uid
   across occurrences, and the master-outside-window read gap. Mitigations: span
   required, `RECURRING_SPAN_UNSUPPORTED`, the bounded recurring-master pass with
   Python expansion; residual risk is exotic RRULEs returned unexpanded (flagged, not
   dropped).
5. **Calendar.app auto-launch.** Every AppleScript call may launch Calendar.app UI on
   the user's machine (RR 4.1). Accepted (matches Mail behavior); documented in the
   operator skill.
6. **Shared osascript lock.** Calendar and Mail tools serialize behind one lock; a slow
   calendar read delays mail tools in the same session. Accepted for 3.10.0 (RR 7.3);
   revisit with live data.
7. **PR #70 version-file conflict.** Trivial but must be resolved by whichever branch
   merges second (section 6.8).
8. **Tool-count regex false positives.** New prose mentioning numbers near the word
   "tools" in required files can trip the blunt claim regex; re-run
   `bash tools/gates/dev-check.sh manifest` after doc edits.
9. **`tools/calendar` package name vs stdlib `calendar`.** Absolute imports keep this
   safe, but any `import calendar` inside the package would bind the stdlib module;
   convention note added in the facade docstring and enforced by review.
10. **Hook generalization regression.** The `check_applescript_compiles.py` edit touches
    a hook that guards Mail scripts too; keep the Mail literal passing in its own tests
    before extending (Wave 0 lands first and alone).
11. **EventKit runloop glue.** The read engine's auth checks are synchronous (safe), but
    `calendar-grant`'s request pump is the one place that can hang; it is CLI-only,
    capped at 30 s, and never reachable from a tool call. De-scope trigger: if the
    EventKit engine cannot be live-verified before release, ship 3.10.0
    AppleScript-only behind the same Protocol seam and move `eventkit.py` to the
    forward queue; no tool signatures change.
