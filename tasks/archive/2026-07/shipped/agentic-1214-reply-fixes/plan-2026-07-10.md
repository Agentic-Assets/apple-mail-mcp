# AGENTIC-1214 reply-body fixes: implementation plan

Branch: `fix/agentic-1214-reply-body-truncation` (off `main` @ 3.10.1).
Owner of merge/commit: the orchestrator. Implementers do NOT commit, push, or
merge, and do NOT run osascript / Mail / the apple-mail CLI. One dedicated
live-verification agent may exercise Mail in draft mode only.

This plan is the verbatim contract. Every file target, constant, error code,
AppleScript fragment, and test below is prescriptive. Where a value is marked
"live-tunable", the live-verification agent may adjust it, but the default ships
if live testing is not available.

Root-cause chain (confirmed in code by phase-1 reports and re-read here):
1. `reply_scripts.py:470` types the ENTIRE `reply_body` with ONE
   `keystroke replyBodyText` call inside the 4-attempt focus guard
   (`_build_reply_native_window_applescript`, lines 428-477). Long bodies drop
   their tail near 320-480 chars (Bug 1, deterministic); short bodies sometimes
   type ALL CAPS (Bug 3, intermittent).
2. `saved_draft_checks.py:59` builds the verifier needle from
   `_first_non_empty_line(reply_body)` only, so a truncated body whose first
   line survived passes verification.
3. The verifier substring search runs through `textOffset`
   (`applescript_snippets.py:37-58`), whose text-item-delimiter comparison is
   case-insensitive by default, so ALL-CAPS drafts also pass.
4. `manage.py` `action="create"` (lines 136-226) never reads `in_reply_to`;
   the reply-like guard `_standalone_compose_thread_warning`
   (`payload.py:117-145`) hardcodes the tool name `compose_email` and never
   explains that create cannot thread.

## A. Bug 1 + Bug 3: chunked native typed insertion

Decision: chunked System Events keystroke with modifier-state hygiene and an
inter-chunk focus re-check. Clipboard stays banned (two prior live reverts:
cross-thread body leak `6bfe72b`, silent body loss `3d8c81a`; doctrine in
`reply_scripts.py` docstring lines 320-337). We do NOT reassign
`content of replyMessage` (flattens the rich quote bar and logo signature, a
prior shipped bug). We keep the native `reply ... with opening window` path.

### A.1 New constant (compose/constants.py, currently 20 LOC)

Add a `TYPING_BOUNDS` dict:

```python
# System Events keystroke throughput bounds for the native reply typed path.
# A single keystroke of the whole reply_body drops its tail near 320-480 chars
# (AGENTIC-1214 Bug 1) and can leak shift-state into ALL CAPS (Bug 3). Type in
# small chunks with a settle delay so Mail's WebKit compose editor keeps up, and
# clear modifier state between chunks. CHUNK_SIZE is well below the observed
# truncation floor; both values are empirically tunable against Mail on the host.
TYPING_BOUNDS = {
    "CHUNK_SIZE": 80,          # characters per keystroke call (live-tunable, floor 1)
    "INTER_CHUNK_DELAY": 0.35, # seconds between chunks; matches the 0.3-0.5 guard delays
}
```

Budget math from the code map: worst realistic successful native run consumes
about 8.1s of the 120s script budget, roughly 112s of headroom. A 1000-char
body at CHUNK_SIZE=80 is about 13 chunks x 0.35s = ~4.5s of typing. A 5000-char
body is about 63 chunks x ~22s. Both land inside headroom; `timeout=240` does
not help because the failure is per-call keystroke coalescing, not the budget
(consistent with the field report).

### A.2 New leaf module: compose/typing_scripts.py

`reply_scripts.py` is 539 LOC; the module budget warns at 600 and CI blocks
growth past baseline (baseline `modules` is empty, so any file that crosses 600
fails). Put the chunk-typing AppleScript builder in a NEW leaf module so
`reply_scripts.py` stays under budget.

`compose/typing_scripts.py` exports one builder:

```python
def build_chunked_typing_handler(*, chunk_size: int, inter_chunk_delay: float) -> str:
    """Return the AppleScript handler that types a reply body in focus-guarded chunks."""
```

It returns this AppleScript handler text (numeric literals interpolated from the
two args; nothing user-derived is interpolated, so no escaping is required). The
handler runs inside the top-level script scope, so it may open its own `Mail`
and `System Events` tells:

```applescript
on typeReplyBodyChunks(bodyText, expectedTitle, derivedTitle)
    set bodyLength to count of characters of bodyText
    if bodyLength is 0 then return "typed"
    -- Clear any lingering modifier state before the first chunk. A truncated or
    -- interrupted prior keystroke pass is the suspected source of Bug 3's leaked
    -- shift state; releasing modifiers here resets it for this typing pass.
    tell application "System Events"
        key up {shift, option, control, command}
    end tell
    set chunkStart to 1
    repeat while chunkStart is less than or equal to bodyLength
        set chunkEnd to chunkStart + {chunk_size} - 1
        if chunkEnd > bodyLength then set chunkEnd to bodyLength
        -- Prefer to end a chunk on a newline so line structure stays intact and
        -- no chunk boundary lands mid-line. Fall back to the hard size boundary
        -- when the window holds no newline.
        if chunkEnd < bodyLength then
            set scanIndex to chunkEnd
            repeat while scanIndex > chunkStart
                set scanChar to character scanIndex of bodyText
                if scanChar is return or scanChar is linefeed then
                    set chunkEnd to scanIndex
                    exit repeat
                end if
                set scanIndex to scanIndex - 1
            end repeat
        end if
        set chunkText to text chunkStart thru chunkEnd of bodyText
        -- Re-verify focus before EACH chunk. Mail's dictionary front window must
        -- still be the adopted reply window. A drift means the user or another app
        -- took focus mid-typing; abort so no partial body is left in a stray window.
        set frontName to "(unset)"
        tell application "Mail"
            try
                set frontName to name of front window
            end try
        end tell
        if frontName is not expectedTitle and frontName is not derivedTitle then
            return "interrupted:" & frontName
        end if
        tell application "System Events"
            tell process "Mail"
                key up shift
                keystroke chunkText
            end tell
        end tell
        set chunkStart to chunkEnd + 1
        if chunkStart is less than or equal to bodyLength then delay {inter_chunk_delay}
    end repeat
    return "typed"
end typeReplyBodyChunks
```

Modifier hygiene: full `key up {shift, option, control, command}` once at handler
entry, then `key up shift` before every chunk keystroke (shift is the documented
ALL-CAPS culprit; the Apple-forums fix released `key up {shift}` after every
character, and per-chunk is the coarse form of that). If the on-edit osacompile
hook rejects the multi-modifier list `key up {shift, option, control, command}`,
fall back to the proven single form `key up {shift}` (compiled and ran in the
cited fix) or split into one `key up` statement per modifier. The implementer
resolves the exact token against the hook before proceeding; do not leave a
parse failure.

Unicode: AppleScript `text i thru j` and `character i` operate on composed
characters, so chunk slicing is grapheme-safe at the AppleScript text level. The
newline scan compares against `return` and `linefeed` only.

## B. Full-body, case-sensitive, above-quote post-save verification

Replace the first-line needle with a whole-body compare of the saved draft
content above the quote, whitespace-flattened and substitution-folded, done
case-SENSITIVELY so ALL-CAPS drafts fail.

### B.1 How the full body reaches the verifier: a second temp file

Decision: a second temp file, not an escaped literal. The original
`mail_reply_*` temp file is gone by the time the verifier runs (AppleScript
`rm -f` at every exit plus the Python `finally`). `reply_body` is still a live
Python string. Writing a fresh `mail_reply_verify_*` temp file and `cat`-ing it
avoids embedding a 1000+ char escaped literal in the verifier script.

In `_verify_saved_reply_draft` (saved_draft_checks.py):

1. Write `reply_body` to `tempfile.NamedTemporaryFile(mode="w", suffix=".txt",
   prefix="mail_reply_verify_", delete=False, encoding="utf-8")`; capture
   `verify_body_temp_path`. Use `compose.tempfile` for the existing patch seam
   consistency (reply.py uses `compose.tempfile.NamedTemporaryFile`).
2. Remove `safe_body_needle = escape_applescript(_first_non_empty_line(reply_body))`.
   Keep `safe_quoted_needle` (it is the short `"wrote:"` marker).
3. In the `tell application "Mail"` block, replace
   `set replyBodyNeedle to "{safe_body_needle}"` with
   `set fullReplyBody to do shell script "cat " & quoted form of "{verify_body_temp_path}"`.
4. After `run_applescript`, unlink `verify_body_temp_path` in a `finally` /
   `suppress(OSError)` block. Also emit a `rm -f` of it inside the verifier
   AppleScript at its single return points is optional; the Python cleanup is the
   backstop and the verifier is one run, so a Python `finally` unlink is
   sufficient. Prefer wrapping the existing `try/except` around
   `run_applescript` so the unlink runs on timeout and error too.

### B.2 Normalization set (applied identically to both sides, case PRESERVED)

Add a `flattenForCompare` handler to the verifier script. It applies, in order:

1. Remove all whitespace: `return`, `linefeed`, `tab`, `space`, and
   non-breaking space (`character id 160`).
2. Fold curly single quotes `character id 8216` and `character id 8217` to `'`.
3. Fold curly double quotes `character id 8220` and `character id 8221` to `"`.
4. Fold en dash `character id 8211` and em dash `character id 8212` to `-`.
5. Fold ellipsis `character id 8230` to `...`.

Use `character id N` forms (not literal Unicode in the .py source) so the
osacompile hook and file encoding are never a factor. Implement each step with
the text-item-delimiter replace idiom:

```applescript
on flattenForCompare(theText)
    set t to theText as string
    repeat with stripDelim in {return, linefeed, tab, space, (character id 160)}
        set AppleScript's text item delimiters to (contents of stripDelim)
        set t to (text items of t) as string
    end repeat
    set t to my foldPair(t, (character id 8216), "'")
    set t to my foldPair(t, (character id 8217), "'")
    set t to my foldPair(t, (character id 8220), "\"")
    set t to my foldPair(t, (character id 8221), "\"")
    set t to my foldPair(t, (character id 8211), "-")
    set t to my foldPair(t, (character id 8212), "-")
    set t to my foldPair(t, (character id 8230), "...")
    set AppleScript's text item delimiters to ""
    return t
end flattenForCompare

on foldPair(theText, fromText, toText)
    set AppleScript's text item delimiters to fromText
    set parts to text items of theText
    set AppleScript's text item delimiters to toText
    set joined to parts as string
    set AppleScript's text item delimiters to ""
    return joined
end foldPair
```

Rationale for folding: the domain report confirms Mail's Substitutions
(smart quotes, dashes) are live by default and it is uncertain whether the
WebKit compose editor rewrites typed straight quotes to curly. Without folding,
a deterministic substitution would fail the case-sensitive compare on a perfect
draft, trigger a needless retype, and end in a false REPLY_BODY_MISMATCH.
Folding these classes removes that failure while preserving case (so ALL CAPS is
still caught). Documented fallback the live agent may switch to if folding proves
insufficient: an alphanumeric-only projection (`[A-Za-z0-9]`, case preserved),
which sidesteps all punctuation substitutions at the cost of dropping
punctuation from the match.

### B.3 Case-sensitive above-quote containment

Replace `replyBodyIsBeforeQuote` (saved_draft_checks.py:93-102) with:

```applescript
on replyBodyAboveQuoteStatus(draftContent, fullReplyBody, quotedNeedle)
    set flatBody to my flattenForCompare(fullReplyBody)
    if flatBody is "" then return "found"
    set flatDraft to my flattenForCompare(draftContent)
    set aboveQuote to flatDraft
    if quotedNeedle is not "" then
        set flatQuote to my flattenForCompare(quotedNeedle)
        set quoteOffset to my textOffset(flatDraft, flatQuote)
        if quoteOffset > 1 then set aboveQuote to text 1 thru (quoteOffset - 1) of flatDraft
    end if
    considering case
        if aboveQuote contains flatBody then return "found"
        if flatDraft contains flatBody then return "after_quote"
    end considering
    return "missing"
end replyBodyAboveQuoteStatus
```

`considering case` makes `contains` case-sensitive (default AppleScript string
compare ignores case). The quote marker is located with the existing case-
insensitive `textOffset` (locating `"wrote:"` does not need case sensitivity).
Keep the `stripLineBreaks` handler in the file only if still referenced; the new
`flattenForCompare` supersedes it for the body check, so remove `stripLineBreaks`
and its lone call if nothing else uses it (grep first; `signatureStatus` uses
`quotedNeedle` offsets on raw `draftContent`, not `stripLineBreaks`).

In `verifyReplyDraft` (line 185-197): read `fullReplyBody` once in the outer
tell (per B.1 step 3) and pass it into `verifyReplyDraft` instead of
`replyBodyNeedle`. Rename the parameter to `fullReplyBody`. The empty-body
shortcut `if replyBodyNeedle is ""` becomes `if fullReplyBody is ""`. The
returned wire tokens are UNCHANGED: `FOUND|...`, `BODY_AFTER_QUOTE|id`,
`BODY_MISSING|id`. Because the tokens and delimiters are unchanged,
`_reply_verification_from_output` (verification.py:46-81) and the whole
`_ReplyDraftVerification` parse layer stay as-is.

Note: this now applies to BOTH the native and object-model reply paths (they
share `_verify_saved_reply_draft`). The object-model path assigns content
directly, so full-body compare passes trivially there; no regression.

### B.4 One automatic delete-artifact-and-retype retry, then REPLY_BODY_MISMATCH

In reply.py, the native draft/open success block (lines 419-459) currently runs
once. Wrap the temp-file write, native script build, `run_applescript`, and
verification in a bounded loop of at most two attempts. Only the NATIVE path
retries (object-model never mismatches). The retry fires only on a body
placement failure (`verification.status in {"body_missing", "body_after_quote"}`)
AND only when a concrete artifact id is known to delete (so we never pile drafts).

Control flow (native, draft/open, `mode_plan.success_text in result`):


`reply`/`forward` commands populate them). Therefore: HARD structured refusal
when `in_reply_to` is passed with `action="create"`. This is the recommended
option because headers cannot be written honestly.
