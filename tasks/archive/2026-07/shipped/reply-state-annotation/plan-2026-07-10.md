# Reply-State Annotation: `was_replied_to` + `has_draft` (plan, 2026-07-10)

Target release: **v3.11.0** on branch `feat/reply-state-annotation` (based on `main` @ `122314e`, v3.10.1).

## Problem

No read/triage tool ever checks Drafts, and "already replied" detection exists in only
three tools, always opt-in (off by default), always via an expensive Sent-mailbox header
scan. With default arguments, `get_needs_response` reports mail the user already answered
as "needs response", and an agent can double-draft a reply that already exists. Research
confirmed (2026-07-10, three-agent audit):

- Mail.app maintains a native read-only `was replied to` boolean per message. It is never
  read anywhere in this codebase. Live probe: correct on real replied threads, marginal
  cost ~15ms/message inside the existing per-message property pass (no new AppleScript
  round trip, no `whose`).
- A bounded Drafts snapshot (subject + first to-recipient, `DRAFT_LOOKUP=75` cap) costs
  ~2s flat per call. Full-header reads on drafts cost ~72ms/draft (+166%), so header
  correlation must be sub-capped (mirror `REPLIED_HEADER_READ_CAP=10`). In-Reply-To is
  readable on saved unsent drafts with roughly 80% hit rate; not a sole signal.
- Skills already instruct duplicate-avoidance but only as manual multi-call patterns;
  no conflicts with making it default, only staleness at six literal call sites.

## Contract (final)

### New per-email JSON fields (always in the row shape)

Tools: `list_inbox_emails`, `search_emails`, `get_email_by_id`, `get_email_by_ids`,
`get_email_thread`, `get_needs_response`, `inbox_dashboard`, `get_inbox_overview`
(recent rows).

- `was_replied_to` (bool, always present): Mail native `was replied to`, read in the same
  per-message AppleScript loop as subject/sender/date. No parameter gates it.
- `has_draft` (true | false | null): a draft reply correlates to this email. `null` means
  the draft scan was skipped or errored (never silently false). Governed by a per-tool
  `include_draft_state: bool = True` escape hatch (True is the default: annotation is
  automatic, per the product requirement).
- Text output modes append `[REPLIED]` / `[HAS DRAFT]` markers to rows.
- JSON responses gain a top-level `draft_scan` object:
  `{"status": "ok" | "error" | "skipped", "scanned": N, "accounts": [...], "error": "..."?}`.

### Draft correlation rule (in `core/reply_state.py`)

A draft matches a candidate email when either:

1. Draft `In-Reply-To`/`References` contains the candidate `internet_message_id`
   (headers read only for the newest `DRAFT_SNAPSHOT_HEADER_CAP = 10` drafts), or
2. normalized subject equality (strip `THREAD_PREFIXES` repeatedly, casefold, trim)
   AND draft first to-recipient equals candidate sender address (case-insensitive)
   AND draft date >= candidate date minus 1 day (a reply draft postdates its source;
   this kills stale same-subject false positives without a magic window).

One snapshot per account per tool call (never per candidate; never
`manage_drafts(action="find")` in a loop). Multi-account fan-out fetches snapshots
lazily per account appearing in results, capped at 5 accounts. Snapshot uses a
Drafts-mailbox name-fallback resolver ("Drafts", "Brouillons", "Entwürfe",
"Borradores"), the fallback treatment Inbox (11 names) and Sent (3 names) already
have and Drafts never got.

### `get_needs_response` (becomes the "what still needs me" tool)

- Default now EXCLUDES rows with `was_replied_to=true` or `has_draft=true`.
- Output reports `skipped_replied_count` and `skipped_drafted_count` (visible, not silent).
- `include_already_replied=True` keeps replied rows (annotated); new
  `include_drafted: bool = False` keeps drafted rows (annotated).
- `check_already_replied` (legacy Sent-header scan) stays as an opt-in extra
  verification layer, default False, docstring reframed: the native flag is now the
  primary signal.
- Fail-open on draft-scan error: nothing is excluded for draft state and
  `draft_scan.status = "error"`.

### `list_inbox_emails` / `search_emails`

- `exclude_replied` keeps its name but now filters on the native flag (no Sent scan
  needed for it). New `exclude_drafted: bool = False`. `flag_replied` accepted for
  compatibility; the canonical field is `was_replied_to` (docstring deprecation note).
  The legacy `already_replied` field remains only where the opt-in Sent-scan path is
  explicitly invoked.

### Explicitly unchanged

- `get_awaiting_reply`: opposite direction (Sent -> did they reply to me); no native
  property exists for that direction; its header scan stays.
- `get_mailbox_unread_counts`, `get_top_senders`: no per-email rows.
- `manage_drafts` / `verify_draft`: single-account, no-fallback Drafts lookup weakness
  stays a roadmap Hardening item (the new resolver is available for that later fix).

## Verification path

Mocked pytest per surface (annotation fields present, default exclusion + skip counts,
correlation rule edge cases, fallback resolver, parser row-shape change). Then live
against the real account: `apple-mail quick-check`, `list_inbox_emails` JSON shows
`was_replied_to` on known replied threads, `get_needs_response` default excludes them
and reports skip counts, `include_*` flags restore them.

## Open caveats (recorded, not blockers)

1. Unconfirmed whether Mail flips `was replied to` when a reply draft is opened/saved
   versus only on send. The precedence rule (a matching draft always reports as
   drafted; "replied with no draft found" is the only pure replied state) keeps agent
   guidance correct either way. Live confirmation needs a disposable message and is
   deferred; skills phrase the guarantee accordingly.
2. In-Reply-To hit rate on drafts is ~80%; subject+recipient+date rule covers the rest.
   Neither alone is sufficient; combined recall is the design.
3. `tasks/todo.md` / `tasks/INDEX.md` rows for this lane are deferred to close-out:
   the pending `chore/tasks-roadmap-refresh` branch rewrites both files and editing
   them here would manufacture conflicts.
